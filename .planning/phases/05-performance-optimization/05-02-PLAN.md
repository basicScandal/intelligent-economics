**Pattern for all three files (PERF-05):**

Each file's render loop currently calls `requestAnimationFrame(self)` unconditionally. Change each to use an `isVisible` boolean gated by IntersectionObserver on the canvas's parent section.

**hero-particles.ts:**

1. After creating the renderer/scene/camera and before the animate loop, add:
   ```typescript
   let isVisible = true;
   const observer = new IntersectionObserver(
     (entries) => {
       isVisible = entries[0].isIntersecting;
       if (isVisible && !animRunning) startLoop();
     },
     { threshold: 0 }
   );
   // Observe the hero section (canvas parent's parent), not the canvas itself
   const heroSection = canvas.closest('section') || canvas.parentElement;
   if (heroSection) observer.observe(heroSection);
   ```

2. Refactor the animate loop to be start/stoppable:
   ```typescript
   let animId: number | null = null;
   let animRunning = false;

   function animate(): void {
     if (!isVisible) {
       animRunning = false;
       return;
     }
     animId = requestAnimationFrame(animate);
     animRunning = true;
     // ... existing frame logic (particle position updates, rotation, render) ...
   }

   function startLoop(): void {
     if (!animRunning) {
       animRunning = true;
       animate();
     }
   }

   startLoop(); // initial start
   ```

   The key insight: when `isVisible` becomes false, the loop self-terminates (no more rAF calls). When IntersectionObserver fires with `isIntersecting: true`, `startLoop()` restarts it.

**morph-engine.ts:**

1. Same IntersectionObserver pattern. Observe `morphSceneEl` (the morph scene wrapper element, already available as a variable).

2. Refactor `mLoop()` identically:
   ```typescript
   let mVisible = true;
   let mAnimRunning = false;

   const mObserver = new IntersectionObserver(
     (entries) => {
       mVisible = entries[0].isIntersecting;
       if (mVisible && !mAnimRunning) startMLoop();
     },
     { threshold: 0 }
   );
   mObserver.observe(morphSceneEl);
   ```

3. Rename internal animate function to use the pattern:
   ```typescript
   function mLoop(): void {
     if (!mVisible) {
       mAnimRunning = false;
       return;
     }
     requestAnimationFrame(mLoop);
     mAnimRunning = true;
     // ... existing frame logic ...
   }

   function startMLoop(): void {
     if (!mAnimRunning) {
       mAnimRunning = true;
       mLoop();
     }
   }

   startMLoop();
   ```

   IMPORTANT: The GSAP ScrollTrigger `onUpdate` callback that triggers `goTo()` still works because ScrollTrigger fires independently of the render loop. The morph animations (GSAP tweens) update `livePos` even when the loop is paused. When the loop resumes, the particles will be at the correct morph position.

**zone-zero.ts:**

1. Same IntersectionObserver pattern. Observe the zone-zero section. The section is already queried via `document.getElementById('zone-zero')` later in the file for analytics. Query it earlier or reuse the `container` parameter passed to `initZoneZero()`.

2. Refactor `animate()` identically:
   ```typescript
   let zzVisible = true;
   let zzAnimRunning = false;

   const zzObserver = new IntersectionObserver(
     (entries) => {
       zzVisible = entries[0].isIntersecting;
       if (zzVisible && !zzAnimRunning) startZZLoop();
     },
     { threshold: 0 }
   );
   // Use the container element passed to initZoneZero, or fall back to section
   const zzSection = container.closest('section') || container;
   zzObserver.observe(zzSection);
   ```

3. The existing `zzObserver` for analytics tracking (CONV-09) must remain separate — it fires once and disconnects. The render-pause observer is persistent. Do NOT merge them. Rename the analytics one to `analyticsObserver` for clarity if both use `zzObserver` name.

**Do NOT change:** Particle counts, color logic, GSAP integrations, mouse handlers, resize handlers, or any UI logic. Only the rAF loop scheduling changes.
